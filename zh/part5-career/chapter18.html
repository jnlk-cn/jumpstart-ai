第16章 入职前30天：快速站稳 ​

本章你能带走什么 ​

恭喜你，终于从"求职者"升级成了"打工人"！

拿到Offer的那一刻，你可能长舒一口气，觉得最难的关卡已经过了。但我必须给你泼一盆冷水：入职才是真正的开始。

很多人以为进了公司就安全了，殊不知前30天才是决定你能不能留下来的关键时刻。想象一下：

  • 第一天到公司，面对陌生的代码库一脸懵，不知道从哪下手
  • 第一个任务布置下来，听得云里雾里，又不敢追问，怕显得太菜
  • 两周过去了，还在熟悉环境，同事已经开始用异样的眼光看你
  • 一个月结束，Leader找你谈话："你这一个月都干了什么？"

如果你不想成为上面任何一个场景的主角，这章就是为你写的。

读完这章你会带走：

  • 入职第一周的自救清单，让你快速进入状态
  • 快速读懂现有代码库的方法论
  • 第一个任务怎么接、怎么交的实战技巧
  • 与团队磨合的沟通策略和AI开发在团队中的定位

好了，系好安全带，我们开始。

16.1 新人入职的自救清单 ​

入职前的准备工作 ​

别等到入职那天才慌慌张张。在正式入职之前，你应该已经做好以下准备：

1. 硬件和软件环境

入职前一两天，你会收到IT部门发来的账号密码、邮箱配置、开发机申请流程等邮件。这些邮件可能很长，但一定要仔细看完。

常见的入职准备清单：

☐ 开发机申请（有些公司需要自己申请，等机器到位可能要几天）
☐ VPN配置（在家办公必备）
☐ GitLab/GitHub权限申请
☐ 代码仓库访问权限
☐ 公司内部文档平台权限（Confluence、飞书文档等）
☐ 通讯工具配置（Slack、企业微信、钉钉、飞书）
☐ 开发环境配置（IDE、Python环境、依赖管理工具）
☐ 熟悉公司代码仓库结构

我的建议是：入职前就把能申请的权限都申请了，能装的软件都装了。第一天上班第一天就开始装环境是很蠢的，白白浪费一上午。

2. 了解你要加入的团队

入职前做功课是基本操作：

  • 去公司官网看团队介绍
  • 搜一下团队的公开技术博客、技术分享视频
  • 如果是上市公司，看看财报里有没有提到AI相关的业务布局
  • 领英上搜一下你未来的同事，了解他们的背景

这些信息能帮你快速理解团队的定位和方向，入职后跟同事聊天也不至于完全摸不着头脑。

3. 心理建设：接受"菜鸟"状态

最重要的一点心理准备：你是一个菜鸟，这是正常的。

新人入职最大的心理障碍是"我不能露怯"、"我不能让同事觉得我很菜"。这种想法害死人。

真实情况是：

  • 所有人都知道你是新人
  • 所有人都预期你会有很多不懂的地方
  • 问问题不会被嘲笑，装懂才会

你的策略应该是主动暴露问题，而不是假装没问题。等别人发现你有问题的时候，你就真的有问题了。

入职第一天的生存法则 ​

入职第一天，核心目标是活下来，别惹事。

1. 准时，甚至提前到

第一天迟到是灾难级别的。我见过有人入职第一天就迟到半小时，给整个团队留下了极差的印象。别给自己挖坑。

2. 少说话，多观察

第一天不要急于表现，疯狂输出观点。你对公司、对团队、对业务一无所知，说的很多话可能是错的，甚至可能得罪人。

观察什么呢？

  • 团队的沟通风格：活跃还是内敛？开会讨论还是单向汇报？
  • 技术氛围：代码评审严格吗？有技术分享吗？
  • 工作节奏：加班多吗？下班时间大概几点？
  • 茶水间八卦：别主动凑上去，但可以竖起耳朵听

3. 认识关键人物

入职第一天通常会有个"认识团队"的环节，可能是一起吃个饭，也可能是HR带你串门拜访。

你需要重点记住这三类人：

  1. 你的直接汇报对象（Leader/Manager）：搞清楚他的工作风格、期望、管理方式
  2. 你的Buddy/导师：如果有公司给你分配了导师，珍惜这个人，他是你最重要的资源
  3. 业务方的关键对接人：了解他们是谁、做什么、可能找你做什么

4. 问正确的问题

入职第一天肯定会有人问你"有什么问题吗"，你可以准备几个好问题：

"咱们团队现在最重要的项目是什么？"
"我入职后第一个任务会是什么？"
"团队的日常工作流程是什么样的？"
"代码规范和Code Review是怎么做的？"
"有没有我入职前应该先看的文档或代码？"

入职第一周的自救清单 ​

入职第一周是建立印象的关键期。以下是你必须完成的事情：

1. 环境必须跑通

☐ 本地开发环境搭建完成（Python版本、依赖包）
☐ 能pull代码、能push代码
☐ 能跑单元测试
☐ 能启动一个最简单的服务
☐ 能连上测试数据库或mock数据
☐ VPN、日志系统、监控平台能正常使用
1

如果一周结束你的环境还没跑通，Leader会认为你连基本的工作能力都没有。

2. 代码必须读过

第一周至少要通读一遍你负责模块的代码。注意是"通读"，不是精读。先把整体结构搞清楚。

  • 这个模块负责什么功能？
  • 有哪些核心类/函数？
  • 数据流向是怎样的？
  • 依赖了哪些外部服务？

3. 会议必须参与

即便你什么都不懂，也要参加团队的所有会议。听不懂没关系，但要让别人看到你在场。

会议中你可以做两件事：

  1. 记录：记下不懂的术语、人名、项目名，会后去查
  2. 提问：会后找你的Buddy/导师问："刚才会上提到的XX是什么？"

4. 产出必须有过

第一周结束，你必须有一个可以展示的产出。不需要是什么大功能，可以是：

  • 一个文档：整理了某个模块的代码结构
  • 一个修复：修了一个简单的bug
  • 一个测试：补充了一个测试用例
  • 一个调研：研究了某个技术方案的可行性

哪怕很小，也比"什么都没干"强一万倍。

16.2 如何快速理解现有代码库 ​

新人最常犯的错误是什么？对着代码库从头看到尾，结果越看越懵。

AI项目的代码库通常有以下特点：

  • 代码量大，少则几千行，多则几万行
  • 依赖复杂，可能用了LangChain、向量数据库、LLM API等多个外部服务
  • 文档可能缺失或不更新
  • 有一些历史遗留的"祖传代码"，没人敢动

想在一周内完全搞懂是不现实的。但你不需要完全搞懂，你需要搞清楚跟你相关的部分。

第四步：追踪关键流程

AI应用的核心流程通常就那么几条：

  1. 用户请求流程：请求→路由→处理→返回
  2. 数据索引流程：文档→切分→向量化→入库
  3. Agent执行流程：用户意图→规划→工具调用→结果整合

找一条最核心的流程，从头到尾读一遍代码。不懂的地方标注出来，会后去问。

1. 用IDE的搜索功能

VS Code或PyCharm的全局搜索（Ctrl+Shift+F）是神器。

当你听说一个功能，比如"RAG检索"，直接搜索"retrieval"或"search"，就能快速定位到相关代码。

2. 从入口文件开始

入口文件通常如下：

16.3 第一个任务：怎么接、怎么交 ​

怎么接任务 ​

新人最怕的事情之一：Leader给你布置任务，你听不太懂，但不敢问。

千万别这样。没听懂装懂，比没听懂去问，后果严重十倍。

接任务的正确姿势：

1. 先听，先记，别急着回应

Leader布置任务的时候，先把话听完。不要中途打断，说"我知道了"或"这个简单"。等Leader说完，你再开口。

2. 用你自己的话复述一遍

Leader：帮我们加一个功能，用户问问题的时候先检索知识库，找不到答案再调用大模型。

你：所以这个需求是说，如果用户问的问题能在知识库里找到匹配的文档，就直接返回文档内容；如果找不到，才去问大模型，对吗？
1
2
3

复述的目的有两个：

  • 确认你理解得对不对
  • 让Leader知道你在认真听

3. 确认范围和优先级

你：明白了。我可以确认几个问题吗？
1. 这个功能需要支持哪些知识库？是现有的还是新建的？
2. "找不到答案"的判断标准是什么？是没有检索结果，还是置信度低于某个阈值？
3. 这个任务的优先级是怎样的？需要什么时候完成？
1
2
3
4

这些问题展示了你的严谨思考，比闷头干活强得多。

4. 主动预估时间

新人往往会低估或高估任务所需时间。我的建议是：

  • 如果你觉得2天能做完，就说"我预计需要2-3天，先做一版给您看看方向对不对"
  • 不要说死，给一点buffer
  • 如果不确定，就说"我先调研一下，半天后给您一个更准确的时间评估"

任务接过来之后 ​

1. 先调研，再动手

拿到任务不要马上写代码。先问自己几个问题：

当前代码库中有没有类似的实现？
有没有可以直接复用的组件？
这个任务涉及哪些模块，需要跟谁对齐？
有没有技术债务需要先处理？
1

调研花1-2天是正常的，不要觉得是浪费时间。调研充分了，写代码反而快。

2. 小步提交，及时同步

不要憋一个大招，两周后才交付。正确的做法是：

Day 1-2: 调研完成，跟Leader对齐方案
Day 3-4: 实现核心逻辑，提交第一个PR
Day 5: Code Review，修复反馈
Day 6-7: 完善功能，测试，提交最终版本

3. 看错误日志，指向XXX

高效求助意味着：

  • 你已经做了自己能做的排查
  • 你有明确的疑问点
  • 你给Leader的是判断题，不是问答题

4. 主动管理Leader的预期

如果发现任务比预想的复杂，或者时间不够用，提前说，不要等到最后一天才说。

我：X总，跟您同步一个情况。之前评估的RAG任务，有个问题我没考虑到：知识库的文档格式比较乱，直接检索效果不好，需要先做文档清洗。

我的建议是分两步走：
第一步先上线基础版本，不做清洗，只支持格式规范的文档
第二步后续迭代，加上文档清洗模块

怎么交任务 ​

1. 交付标准要提前明确

在接任务的时候就问清楚：

text
"交付的标准是什么？是能跑就行，还是需要达到某个准确率？"
"需要写测试吗？测试覆盖率要求多少？"
"需要更新文档吗？"

2. 提交PR要有好的描述

Code Review是团队协作的重要环节。你的PR描述应该包含：

markdown

这个PR做了什么

## 实现方案
用了什么技术，怎么实现的

## 测试情况

## 影响范围
改了哪些文件，可能影响什么功能

如果有UI或输出变化，附上截图或示例

3. 准备好被问三个问题

Code Review的时候，Reviewer通常会问：

  1. 为什么这么做？ - 你需要解释设计思路
  2. 考虑过XX情况吗？ - 你需要说明边界情况
  3. 为什么用A而不是B？ - 你需要说明方案取舍

被问到这些问题是正常的，说明Reviewer在认真看你的代码。不要慌，坦诚回答就好。

4. 收到反馈要积极响应

Code Review的反馈要认真对待：

  • 有问题就改，不要argue
  • 如果觉得建议不合理，可以说"我理解您的考虑，我的想法是XX，您看这样可以吗？"
  • 改完之后主动通知Reviewer

16.4 与团队磨合：AI开发在团队中的定位 ​

AI开发在团队中的典型角色 ​

进入一个公司后，你需要搞清楚：AI开发在你的团队里是什么定位？

场景1：AI是核心产品

如果你的公司本身就是做AI产品的（AI客服、AI助手、AI写作工具等），那AI开发就是核心岗位。

这种情况下：

  • 你会被赋予重要任务
  • 技术话语权比较大
  • 但压力也大，AI效果不好直接关系公司生死

场景2：AI是业务赋能工具

如果你的公司用AI来提升现有业务效率（智能客服、智能审核、智能推荐等），那AI开发更多是支撑角色。

这种情况下：

  • 你的工作要服务于业务目标
  • 技术上可能受限于业务方的时间窗口
  • 需要跟业务方密切配合

场景3：AI是探索性项目

有些公司把AI当成战略探索，还没有形成稳定的产品形态。

这种情况下：

  • 变动可能很大，方向可能随时调整
  • 需要你有比较强的自我驱动能力
  • 也意味着更大的发挥空间

搞清楚自己处于哪个场景，才能调整自己的心态和工作方式。

找到你的"生存空间" ​

新人入职，最怕的是两件事：太边缘和太突出。

太边缘：没人理你，没有任务，每天无所事事。这种情况要么是团队不重视AI，要么是你没找到切入点。

太突出：锋芒毕露，抢了别人的风头，可能得罪人。新人要低调，先站稳再求发展。

正确的策略是找到一个"既能贡献价值，又不威胁别人"的切入点。

1. 从"脏活累活"切入

每个团队都有一些没人愿意干的活：

  • 写文档
  • 补充测试
  • 优化CI/CD流程
  • 整理日志和监控

这些活技术含量不高，但团队很需要。主动承担这些，能快速建立"靠谱"的印象。

2. 从"增量"而不是"存量"切入

不要一上来就改核心模块，那是祖传代码，没人希望新人乱动。

找一个新的、小的需求来做：

  • 帮业务方加一个新功能
  • 调研一个新技术方案
  • 做一个小优化

做出成果后再逐步深入核心模块。

3. 建立"问题解决者"的人设

新人在团队里最快的成长方式是什么？成为那个能解决问题的人。

当别人遇到AI相关的问题时，如果你能帮忙解决，你就有了价值。

同事：帮看看这个Prompt为什么效果不好？
你：我觉得可能是Few-shot的示例不太合适，我帮你调一下试试。

同事：这个Embedding服务最近响应很慢，你知道怎么回事吗？
你：我查了一下日志，发现是这个接口的并发量上去了，我来优化一下缓存策略。

能解决问题的人，永远是团队最需要的人。

跨部门协作的技巧 ​

AI开发经常需要跟产品、运营、业务等多个部门打交道。跨部门沟通是技术人的短板，但必须补齐。

1. 理解对方的语言

产品经理说"我要一个能理解用户意图的AI"，技术人应该翻译成：

  • 意图识别（NLU）
  • 多轮对话管理
  • 知识库检索

运营说"这个回答不够专业"，技术人应该翻译成：

  • Prompt需要优化
  • 或者知识库质量需要提升
  • 或者模型需要微调

把业务语言翻译成技术语言，把技术语言翻译成业务语言，这是AI开发者的核心能力。

2. 管理对方的预期

业务方往往对AI有不切实际的期望：

  • "AI能读懂所有文档吗？"
  • "AI能100%准确回答问题吗？"
  • "AI能像人一样思考吗？"

你要做的不是否定AI，而是设定合理的边界：

"目前AI在XX场景下效果比较好，准确率大概在85%左右；在XX场景下效果不太稳定，需要人工介入。"
"AI的优势是XX，但也有局限性，比如XX。我们建议的方案是XX。"

3. 记录和留痕

跨部门沟通最怕的是"说了不算"。重要的事情一定要有记录：

  • 开会结论要发会议纪要
  • 需求变更要走需求评审流程
  • 技术方案要对齐并记录

这样既保护自己，也保护团队。

持续建立信任 ​

在团队中立足，不是一天两天的事情。你需要持续建立信任。

1. 承诺的事情必须做到

新人最容易犯的错误是：Leader问"这个能搞定吗"，回答"没问题"，结果搞不定。

宁可保守承诺，也不要过度承诺。

如果说"我尽量"，就要真的尽量。 如果说"这个有点难，我试试"，最后没成功也是可以接受的。

说到做到是建立信任的最快方式。

2. 主动汇报进度

✅ 承担责任，说明自己学到了什么

❌ 同一个错误犯两次

本章行动清单 ​

  1. 入职前完成准备工作

    硬件软件环境配置、公司资料了解、心理建设。入职第一天就开始装环境是最低效的做法。

  2. 入职第一周完成四个"必须"

    环境必须跑通、代码必须读过、会议必须参与、产出必须有过。这四件事是你站稳脚跟的基础。

  3. 用"从文档到代码"的路径理解代码库

    先找文档→再跑起来→然后画架构图→最后追踪关键流程。不要对着代码从头看到尾，效率太低。

  4. 掌握快速定位代码的技巧

    用IDE搜索、从入口文件开始、关注配置文件、善用Git历史。这些技巧能帮你节省大量时间。

  5. 接任务时复述确认，不懂就问

    用自己的话复述任务内容，确认范围和优先级，主动预估时间。没听懂装懂比没听懂去问后果严重十倍。

  6. 任务执行中保持小步提交和及时同步

    不要憋大招，每完成一个小节点就同步进度，遇到问题高效求助。

  7. 找到自己的"生存空间"

    从脏活累活切入、从增量而不是存量切入、建立一个"问题解决者"的人设。

  8. 建立跨部门协作能力

    理解对方的语言、管理对方的预期、记录和留痕。这些是AI开发者的核心竞争力。

  9. 持续建立信任

    承诺的事情必须做到、主动汇报进度、承认错误和不足。说到做到是建立信任的最快方式。

  10. 记住：你是一个菜鸟，这是正常的

    新人不需要什么都懂，但需要愿意学、主动问、快速成长。
