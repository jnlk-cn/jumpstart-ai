#### 14.1 AI开发岗简历的写法：项目驱动而非技能罗列

###### 简历是卖给HR的，不是写给自己的

先泼一盆冷水：**你的简历不是给你自己看的。**

很多人写简历的逻辑是"把我会的都列上去"，这完全是错的。HR每天看几百份简历，平均一封只花30秒。你的任务是在30秒内让她觉得"这人值得聊一聊"。

30秒，记住这个时间。

###### 传统简历写法 vs AI岗位简历写法

**传统简历长这样：**

```
姓名：张三
学历：本科 计算机科学
技能：Python、Java、MySQL、Redis、Docker、K8s...
经历：
2021-2024 XX公司 后端开发
- 负责公司后台系统开发
- 参与微服务架构改造
- 优化数据库查询性能
```

**AI岗位应该长这样：**

```
姓名：张三
学历：本科 计算机科学
核心标签：RAG实战派 | Agent开发 | 从0到1落地经验

项目经历：

【RAG知识库问答系统】（2024.03-2024.06）
公司：XX科技 | 角色：核心开发
项目背景：为公司200+员工搭建内部知识库，解决客服培训周期长的问题
技术方案：LangChain + Chroma + BGE-M3 embedding + FastAPI
核心工作：
- 设计文档切分策略，对比固定分块vs语义分块效果，最终采用递归字符分块+标题锚点，准确率提升15%
- 实现混合检索：向量检索+BM25关键词检索，重排序后Top5准确率达89%
- 优化Prompt设计，采用CRISPE框架，模型幻觉率降低60%
- 部署上线，支持50+并发，日均处理2000+请求，P99延迟<2s
技术栈：LangChain / Chroma / BGE-M3 / FastAPI / Docker

【智能客服Agent】（2024.01-2024.03）
项目描述：基于ReAct架构的多轮对话客服，支持订单查询、退换货、投诉处理
核心贡献：
- 设计Agent Tool体系，封装6个业务工具（查订单、退货、查库存等）
- 实现Function Calling流式响应，首token延迟<500ms
- 实现人工兜底机制，当AI置信度<0.7时自动转人工，投诉率下降40%
技术栈：LangChain / GPT-4 / Function Calling / Redis / WebSocket
```

看到区别了吗？

**传统写法展示的是"你会什么工具"，项目驱动写法展示的是"你解决过什么问题"。**

在AI领域，工具会过时（LangChain 0.x到1.0变化巨大），但解决问题的思路是通用的。面试官更想看到的是你**怎么用工具解决真实问题**，而不是你会不会用某个工具。

###### 简历的核心公式

分享一个写AI岗位简历的核心公式：

```
项目 = 背景 + 技术方案 + 核心挑战 + 你的贡献 + 数据结果
```

**背景**：为什么做这个？（业务价值）
**技术方案**：用了什么技术？（专业性）
**核心挑战**：遇到了什么难题？（深度）
**你的贡献**：你具体做了什么？（可验证）
**数据结果**：带来了什么效果？（说服力）

按这个公式写出来的项目经历，HR和面试官都会多看两眼。

#### 14.2 GitHub Demo搭建：让代码替你说话

###### 为什么GitHub比简历更重要

说句不好听的：**简历是可以包装的，但GitHub是骗不了人的。**

面试官看完简历，问你"这个项目是怎么实现的"，你可以吹得天花乱坠。但如果让你现场打开代码讲解，水平高低立判。

GitHub的作用：
1. **证明你不是培训班出来的**：有真实的代码仓库
2. **展示工程能力**：代码规范、README质量、测试覆盖率

###### 让Demo脱颖而出的技巧

**技巧1：加评估指标**

不只是"能跑"，要展示"跑得好"。

```python
##### 评估RAG系统的检索质量
def evaluate_retrieval(test_cases: List[dict]):
    """评估检索准确率"""
    results = []
    for case in test_cases:
        retrieved = rag.retrieve(case["query"])
        relevant = set(retrieved) & set(case["relevant_docs"])
        precision = len(relevant) / len(retrieved) if retrieved else 0
        recall = len(relevant) / len(case["relevant_docs"]) if case["relevant_docs"] else 0
        results.append({"precision": precision, "recall": recall})
    
    # 打印评估结果
    avg_precision = sum(r["precision"] for r in results) / len(results)
    avg_recall = sum(r["recall"] for r in results) / len(results)
    print(f"Avg Precision: {avg_precision:.2%}")
    print(f"Avg Recall: {avg_recall:.2%}")
    return results
```

#### 14.3 技术博客与作品展示

###### 为什么技术博客很重要

说实话，写技术博客对找工作的帮助可能比你想的大得多：

1. **证明表达能力**：技术好的人很多，但能写清楚的人不多
2. **沉淀知识**：写一遍比看十遍记得牢
3. **建立个人品牌**：面试官搜你的名字，看到你的文章会加分
4. **面试加分项**：很多面试官会问你"写过什么技术文章吗"

###### 博客平台选择

| 平台 | 优点 | 缺点 |
|------|------|------|
| 掘金 | AI技术氛围浓，流量大 | 同质化严重 |
| CSDN | 老牌，SEO好 | 质量参差不齐 |
| 知乎 | 影响力大 | 技术内容分散 |
| 微信公众号 | 深度读者 | 传播有限 |
| Medium | 国际影响力 | 国内访问不便 |
| 个人博客 | 品牌专属 | 需要自己引流 |

**我的建议**：掘金 + 个人博客双轨走。掘金拿流量，个人博客建品牌。

###### 什么样的文章值得写

**高价值文章类型**：

**1. 实战复盘型**（最受欢迎）
```
《我用RAG做了一个XXX，从0到1的全过程》
《跑了10个开源RAG项目后，我总结了这些坑》
《从0到1搭一个客服Agent，我踩过的坑》
```
这类文章有过程、有数据、有反思，面试官最爱看。

**2. 深度解析型**
```
《RAG的召回率为什么上不去？我从这5个方面排查》
《从LangChain 0.x到1.0，我趟过的升级之路》
《混合检索为什么比纯向量检索好？实验数据说话》
```
这类文章展示深度思考能力。

**3. 工具测评型**
```
《5款向量数据库横评：Chroma/Qdrant/Milvus/Pinecone/Weaviate》
《Embedding模型选型：中文场景哪家强？》
《RAG评估工具大盘点》
```
这类文章实用性强，容易获得收藏。

**4. 避坑指南型**
```
《RAG开发中最容易踩的5个坑》
《Agent开发新手常犯的7个错误》
《上线LLM应用前必须检查的10件事》
```
这类文章解决实际问题，传播性好。

###### 写作套路：让文章有人看

**开头三板斧**（吸引点击）：
```
开头不要写"最近学习了XXX"，直接写痛点：
- "为什么你的RAG系统召回率总是上不去？"
- "从0到1搭Agent，我花了3天踩了20个坑"
- "面试官问我RAG原理，我只答对了一半"
```

**正文结构**（逻辑清晰）：
```
1. 背景/问题：先说清楚遇到了什么问题
2. 分析过程：你是怎么排查的，试了哪些方法
3. 解决方案：最终怎么解决的
4. 效果数据：有数据才有说服力
5. 总结反思：学到了什么，下次怎么改进
```

**结尾升华**（引导互动）：
```
不要写"以上就是我的分享"，可以写：
- "你们有没有遇到过类似的问题？评论区聊聊"
- "如果你觉得有帮助，欢迎点赞收藏"
- "下期讲XXX，想看的评论区告诉我"
```

###### 把博客变成面试加分项

写完文章不算完，要让文章发挥价值：

**技巧1：在简历里加博客链接**
```
#### 加分项
📝 技术博客：https://juejin.cn/user/yourid（50+篇原创，10万+阅读）
```

**技巧2：在GitHub README里加博客专区**
```markdown
#### 📝 Latest Blog Posts
- [我的RAG实战复盘](link) - 2024.06
- [Agent开发避坑指南](link) - 2024.05
```

**技巧3：主动分享到技术社区**
- 掘金、CSDN、知乎专栏
- 微信技术群、知乎问答
- V2EX、Telegram群组

#### 14.4 不同级别的差异化定位

###### 应届生/转行者（0-1年）：突出潜力

**核心策略：展示学习能力和动手能力**

这个阶段你没有工作经验，面试官也知道。所以他看的是：**你有没有潜力，能不能快速上手**。

**简历定位**：
```
标题：AI应用开发学习者 / RAG技术爱好者
关键词：学习能力强 / 自驱 / 热爱技术 / 快速成长
```

**GitHub定位**：
- 3-5个完整项目（哪怕是跟着教程做的）
- 每个项目要有README、运行说明、效果截图
- 至少一个项目有自己优化/改进的痕迹

**博客定位**：
- 写5-10篇学习笔记/实战复盘
- 重点展示"我学到了什么"和"我遇到了什么坑"
- 不要求深度，但要求真实

**面试话术**：
```
"我确实经验不多，但我做了X个项目，看了Y篇文章，
对这个领域有比较系统的理解。入职后我可以快速上手。"
```

###### 初级工程师（1-3年）：突出独立能力

**核心策略：展示能独立负责一个模块**

这个阶段你应该有1-2个能拿出手的项目了。要展示的是"给我一个任务，我能独立搞定"。

**简历定位**：
```
标题：AI应用开发工程师 / RAG系统开发者
关键词：独立负责 / 落地经验 / 问题解决 / 工程化
```

**GitHub定位**：
- 2-3个核心项目，每个都要有技术深度
- 展示你遇到的问题和解决方案
- 加评估数据、对比实验

**博客定位**：
- 3-5篇实战复盘/深度解析
- 展示你的思考过程和解决问题的方法
- 可以写一些技术对比/选型文章

**面试话术**：
```
"在这个项目里，我是独立负责RAG模块的。
从方案设计到落地到优化都是我一个人做的。
遇到了XX问题，我是这么解决的..."
```

###### 中级工程师（3-5年）：突出业务价值

**核心策略：展示能给业务带来价值**

这个阶段技术已经不是最大问题，面试官更关心的是"你能不能给公司带来价值"。

**简历定位**：
```
标题：高级AI应用开发工程师 / AI技术负责人
关键词：业务理解 / 技术选型 / 团队协作 / 业务增长
```

**GitHub定位**：
- 1-2个展示系统设计能力的项目
- 展示架构设计、技术选型的思考
- 可以有一些架构图、系统设计文档

**博客定位**：
- 2-3篇系统设计/架构相关的文章
- 可以写业务落地案例、ROI分析
- 展示你对业务价值的理解

**面试话术**：
```
"这个AI项目上线后，客服团队的人效提升了40%，
AI独立处理率达到65%，每月节省人力成本XX万。
我认为AI应用开发的核心是解决业务问题，
技术只是手段。"
```

###### 资深工程师（5年+）：突出影响力

**核心策略：展示能影响团队/公司/行业**

这个阶段你要展示的是"我能影响更多人，创造更大价值"。

**简历定位**：
```
标题：AI架构师 / 技术专家 / Tech Lead
关键词：架构设计 / 技术规划 / 团队建设 / 技术影响力
```

**GitHub定位**：
- 开源贡献/开源项目
- 技术标准/规范制定
- 团队内部工具/框架开发

**博客定位**：
- 行业洞察/趋势分析
- 方法论总结
- 布道/分享

**面试话术**：
```
"过去一年，我主导了公司AI基础设施的建设，
搭建了RAG平台，赋能X个业务线。
同时带了一个3人的AI开发小组，
帮助他们从传统开发转型到AI开发。"
```

###### 简历中的"差异化"关键词

根据你的定位，在简历中使用差异化关键词：

**潜力型关键词**（应届生/转行）：
```
✅ 学习能力强、热爱技术、自驱、动手能力
✅ 快速成长、积极主动、善于总结
❌ 但不要写：精通、专家、资深
```

**独立型关键词**（1-3年）：
```
✅ 独立负责、从0到1、落地经验
✅ 问题解决、工程化、稳定可靠
❌ 不要写：架构设计、团队管理
```

**业务型关键词**（3-5年）：
```
✅ 业务理解、价值导向、技术选型
✅ 团队协作、跨部门沟通、数据驱动
❌ 不要写：战略规划、技术决策
```

**影响型关键词**（5年+）：
```
✅ 技术规划、架构设计、团队建设
✅ 技术影响力、跨团队协作、业务增长
❌ 不要写：刚入门、学习中
```
